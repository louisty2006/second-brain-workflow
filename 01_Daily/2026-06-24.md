## Progress

### VibeCode Translator — MVP2.1

整天都在解同一個問題：`Cmd+Ctrl+E` 在 CLI 下完全正常，打包成 `.app` 後完全無反應。

**走過三個方向：**

1. **pynput (CGEventTap)** → 失敗。需要 Accessibility 權限，但 ad-hoc 簽名每次 rebuild 都換 binary hash，TCC 當作新 app → 授權永遠失效。

2. **NSEvent.addGlobalMonitorForEventsMatchingMask** → 失敗。同樣需要 Accessibility，同樣問題。過程中還修了三個 app 死機 bug（runModal 阻塞主線程、runloop 前註冊、Python GC 清掉 handler）。加入診斷 log 後看到 `accessibility=False`，確認根本原因。

3. **Carbon RegisterEventHotKey via ctypes** → 最終成功。完全不需要 Accessibility 權限。但撞了三個 ctypes ARM64 指標截斷 crash：
   - `restype` 預設 `c_int`（32-bit）截斷 64-bit 指標
   - CDLL 每次取屬性都是新 object，restype 設在 A object、call 的是 B object
   - Python int 傳入 ctypes 沒設 argtypes → 當 `c_int` 截斷

   靠崩潰報告暫存器值（`x0 = 0x191d080` vs 正確 `0x60000191d080`）定位每個截斷點。

**最終結果：**
- Hotkey 偵測：Carbon（免 Accessibility）
- 文字擷取：Cmd+C 模擬（首次缺權限時彈提示 + 自動開系統設定）
- 推上 `mvp2-carbon-hotkey` branch，version 2.1.0
- 新增 `DEVLOG.md` 完整記錄整個 debug 歷程（含 ASCII 流程圖）

## Raw Thoughts
ad-hoc 簽名是個隱形陷阱。TCC 看起來授權了，實際上每次 rebuild 都失效。真正的解法是繞開 Accessibility，不是修簽名。Carbon API 是正確答案，但 ctypes 在 ARM64 的 64-bit 指標行為需要非常小心。
